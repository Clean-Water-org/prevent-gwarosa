# 게임 흐름 통합 플랜 — 하루 루프 (09:00 ~ 18:00)

> 메인화면과 미니게임이 따로 노는 현재 상태를, **기획서의 하루 타임라인**대로
> `메인 → 미니게임 → 메인 → … → 엔딩`으로 잇는 통합 작업 플랜.
> 참고: `(최종)기획서_내용_정리.pdf`(타임라인), `notion-brief.md`, `CLAUDE.md`(게임 흐름).

## 1. 확정 사항 (결정 완료)

| 항목 | 결정 |
|---|---|
| 미니게임 라운드 수 | **5라운드** (기획서·notion 기준) |
| 미니게임 선택 | **3종 각각 최소 1회 보장 + 직전과 연속 중복 금지** (이메일/회의/보고서, 랜덤) |
| 메인 → 미니게임 전환 | **시간 도달 시 자동** (메인 체류 시간 경과 후 자동 진입) |
| 업무량 0 달성 | **즉시 칼퇴** (조기 종료, 그 시각 기준 S/A/B 등급) |
| 단계별 게임시간 | 메인 **35분**(고정) / 미니게임 **실제 소요 시간**(최대 60분) / 점심 **60분** |
| 점심 | 라운드 2 직후(≈12:00) |
| 시간 매핑 | **1 실초 = 1 게임분**. 메인 35초 고정 / **미니게임은 실제 소요 초만큼** 게임분 진행 (빨리 깨면 적게, `MAIN_CLOCK_TICK_MS=1000`) |
| 시간 모델 | 고정 스케줄 골격 + **미니게임 소요에 따라 종료 시각 변동** (빨리 깰수록 일찍 퇴근 → 더 좋은 등급). 이벤트는 시간 비소비. 야근 = 18:00에 업무량 > 0 |
| 오프닝/마감 | 09:00(20)=출근(commute)로, 18:00(30)=엔딩 연출로 흡수. 스케줄은 **09:00 메인부터** |

## 2. 기획서 타임라인 (메인 35 / 미니 60 / 점심 60)

| 누적 시각 | 단계 | 소비(분) |
|---|---|---|
| 09:00 | 라운드1 — 메인 | +35 |
| 09:35 | 라운드1 — 미니게임 | +60 |
| 10:35 | 라운드2 — 메인 | +35 |
| 11:10 | 라운드2 — 미니게임 | +60 |
| 12:10 | **점심** (≈12:00) | +60 |
| 13:10 | 라운드3 — 메인 | +35 |
| 13:45 | 라운드3 — 미니게임 | +60 |
| 14:45 | 라운드4 — 메인 | +35 |
| 15:20 | 라운드4 — 미니게임 | +60 |
| 16:20 | 라운드5 — 메인 | +35 |
| 16:55 | 라운드5 — 미니게임 | +60 |
| 17:55 | **마감 판정 → 엔딩** | — |

> 위 표는 **각 미니게임을 60초 꽉 채워 깬 최악의 경우**(합계 ≈ 535분 → ~17:55). 미니게임을 빨리 깨면 그만큼 시계가 덜 가서 **퇴근이 더 빨라진다**(예: 미니마다 30초면 미니 합계 150 → 종료 ~16:25, 더 좋은 등급).
> 게임시간 누적: 메인 +35(고정) · **미니게임 +실제 소요초** · 점심 +60. HUD 시계에 표시된다.

## 3. 상태 모델

기존 `state`에 흐름 제어용 필드를 명확히 한다.
- `gameMinute`: 09:00(540) → 18:00(1080). 시작은 8:58(출근 컷신 후 9:00).
- `minigameRound`: 0~4 (5라운드). 미니게임 완료마다 +1.
- `phase`(또는 `scene`): `main` | `minigame` | `lunch` | `ending`.
- `lastMiniId`: 직전 미니게임 id (연속 중복 회피용) — **신규**.
- `flags.handoverGuideSeen`: 첫 메인화면 가이드 1회 (기존).

## 4. 흐름 로직

```
title → setup → commute → [메인 루프] → ending

[메인 루프]  (5라운드)
  ┌────────────────────────────────────────────┐
  │ 메인화면 진입                                  │
  │  · 첫 진입(라운드1): 인수인계 가이드 1회         │
  │  · 채팅/이벤트 처리 (기존 main-work.js)         │
  │  · 메인 체류 타이머: 일정 시간 경과 → gameMinute +35 │
  │  · 타이머 만료 시 → 미니게임 자동 진입           │
  ├────────────────────────────────────────────┤
  │ 미니게임 (이메일/회의/보고서 중 직전≠랜덤)        │
  │  · 플레이 → 결과(success/partial/fail)         │
  │  · 스탯·업무량 반영 + gameMinute +60           │
  │  · minigameRound += 1, lastMiniId 갱신         │
  ├────────────────────────────────────────────┤
  │ 라운드 종료 처리                                │
  │  · checkEnding() → 엔딩 조건이면 ending         │
  │  · round==2 직후 → lunch(점심, +60) → 메인 복귀 │
  │  · round>=5 → 마감 판정(ending)                │
  │  · 그 외 → 메인화면 복귀(다음 라운드)            │
  └────────────────────────────────────────────┘
```

### 메인 → 미니게임 (시간 자동 전환)
- 메인화면 진입 시 **체류 타이머** 시작. **1실초=1게임분**이므로 메인 35분 = **실시간 35초**.
- 매 초 `gameMinute += 1` 누적(`advanceGameMinute`), **메인 35분이 경과하면** `scene = "minigame"`로 자동 전환.
- 채팅/이벤트는 체류 시간 동안 등장 (기존 `startChatNotifications` 유지).
- 현재 코드: `startMainClock`(1초 틱)이 이미 있으나 `actions.advanceGameMinute`가 main.js에 없어 **표시만 되고 state 미반영** + **미니게임 트리거 없음** → 5-2에서 연결.

### 미니게임 선택 (3종 보장 · 연속 중복 금지)
현재 `index.js`의 고정 `ROTATION` → 아래 규칙으로 5라운드 순서를 만든다 (게임 시작 시 1회 생성 권장).

1. **1~3라운드**: `["email","meeting","report"]`를 **셔플** → 3종이 각각 1회씩, 서로 달라 연속 중복도 없음.
2. **4~5라운드**: **직전 라운드와 다른 것** 중 랜덤.
3. 경계 보장: 4라운드 ≠ 3라운드, 5라운드 ≠ 4라운드.

- 보장되는 조건: **미니게임 1·2·3 모두 최소 1회 등장**, **같은 미니게임이 바로 연달아(예: 1→1) 나오지 않음**.
- 예시 순서: `회의 → 이메일 → 보고서 → 회의 → 보고서` (3종 다 등장, 연속 중복 없음).
- devMode일 때는 `devGameId` 우선(현행 유지).

### 종료 판정 (시간·라운드·즉시)
- **즉시 엔딩**(언제든): 업무량 ≤ 0 → 칼퇴(success), 스트레스 ≥ 100 → 당일퇴사(quit), 체력 ≤ 0 → 과로사(overwork).
- **마감 엔딩**: 라운드 5 완료 또는 `gameMinute ≥ 18:00` → 업무량 ≤ 0이면 칼퇴, 아니면 야근(overtime).
- 칼퇴 등급(S/A/B)은 종료 시점 `gameMinute` 기준(`endings.js`의 `gradeOf`).

## 5. 구현 단계

각 단계 끝나면 멈추고 검토.

### 5-1. 미니게임 순서 생성 (`index.js`, `state.js`)
- `ROTATION` 고정 → **순서 생성 함수**로 교체:
  - 1~3라운드 = `["email","meeting","report"]` 셔플 (3종 보장)
  - 4~5라운드 = 직전과 다른 것 랜덤
  - 게임 시작 시 `state.miniOrder`(길이 5)에 미리 저장하거나, 매 라운드 `lastMiniId` 기준으로 즉석 선택
- 검증 포인트: 3종 모두 포함 + 인접 중복 0 (헤드리스 시뮬레이션)
- 라운드 종료 조건을 `minigameRound >= 5`로 통일

### 5-2. 메인화면 시간 진행 + 자동 트리거 (`main.js`, `main-work.js`)
- `main.js`에 **`advanceGameMinute(n)`** 액션 추가 → `state.gameMinute += n` 후 즉시 엔딩 체크. actions로 전달.
- `main-work.js` `startMainClock`이 이 액션을 호출하도록 연결 (현재는 미연결로 표시만 됨)
- 메인 누적 35분(=35초) 경과 시 `scene = "minigame"`로 **자동 전환**
- 첫 라운드는 인수인계 가이드 종료 후 타이머 시작 (기존 `handoverGuideSeen` 흐름)
- 채팅 알림 시스템과 공존, 씬 전환 시 타이머·패널 정리 (기존 `cleanupMainWorkSystems`)

### 5-3. 라운드 종료·점심·복귀 + 미니게임 실제 소요 시간 (`index.js`, 미니게임들)
- **미니게임 게임시간 = 실제 소요 초**: `GAME_DELTAS`의 고정 `gameMinute: 60` 제거 →
  각 미니게임이 `usedSec`(=60−남은시간)을 `applyResult`로 전달, `applyMiniResult`가 `gameMinute += usedSec`
  - 적용 대상: meeting/report(현재 `usedSec` 추적 중) + email(`minigame.js` 자체 경로)도 동일
- `applyMiniResult`: round 2 직후 lunch(≈12:00, +60), round ≥ 5 → ending, 그 외 → main
- 미니게임 결과 반영 후 `checkEnding`으로 즉시 엔딩 우선 처리

### 5-4. 종료/엔딩 판정 통합 (`state.js`)
- `checkEnding`: 업무량 0 즉시 칼퇴, 18:00 마감 야근/칼퇴 유지·정리
- 등급은 `endings.js gradeOf(gameMinute)` 사용 (이미 구현)

### 5-5. HUD 시계 일관화
- 메인·미니게임·엔딩의 `gameMinute` 표시를 `formatTime`으로 통일 (하드코딩된 "09:05" 등 제거)

### 5-6. 통합 테스트
- 09:00~18:00 한 바퀴: 메인↔미니게임 5라운드 + 점심, 시각 누적 확인
- 즉시 엔딩(업무량0/스트레스100/체력0)과 마감 엔딩(18:00) 각각 도달 확인

## 6. 검토 반영 (기획서 + 현재 구현 대조)

- **시간 모델 = 고정 스케줄 골격 + 미니게임 실제 소요 반영**: 이벤트는 게임시간 비소비. 메인은 35분 고정, 미니게임은 **실제 소요 초만큼** 진행 → 빨리 깰수록 일찍 퇴근(더 좋은 등급). 최악(미니 60초 풀) 시 ~17:55 도달. 야근 = 18:00에 업무량 > 0. (기획서 ~684분은 '업무량 압박' 개념)
- **시간 매핑 1:1 확정**: `MAIN_CLOCK_TICK_MS=1000`(1실초=1게임분). 메인 35초 / 미니게임은 실제 소요 초.
- **메인 HUD는 이미 `state.stats` 반영**(하드코딩 아님) → 5-5는 시계 표기 일관화 위주.
- **업무량 감소 경로 존재**: 상사 업무 "답장" = workload −5. 미니게임(success −20)과 합쳐 17:00 이전 칼퇴(S) 도달 가능.
- **오프닝/마감 버퍼**: 09:00(20)=출근(commute) 흡수, 18:00(30)=엔딩 연출 흡수. 스케줄은 09:00 메인부터.

## 7. 튜닝 포인트 (구현 후 조정)

- **업무량 밸런스**: success −20 × 5 = −100(R5에 0 → B등급). 더 이른 S/A등급은 메인 답장(−5) 누적·미니게임 성공률에 좌우 → 수치 미세조정 가능
- **점심**: 라운드 기반(2회 후) 채택. 필요 시 시간 기반(12:00 도달)으로 변경 가능
- **메인 체류 35초**가 채팅 처리에 충분한지(첫 채팅 3초·이후 15~30초 간격) — 길이 조정 여지
